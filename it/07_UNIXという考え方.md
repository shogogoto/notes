## 0.
  OSは創造者の哲学を身にまとって現れる
    ex. Mac 高度に視覚に訴えるオブジェクト指向のUI 「ほら、あなたのすぐ目の前に」
    ex. DECのOpenVMS ユーザーはPCを恐れていて少数の強力な選択肢に限定
    ex. UNIX「何をしているか分からないのなら、ここにいるべきでない」
      -> 初期のUNIXは支持を得ることに失敗
      初期は大学やハッカーのおもちゃでビジネス目的では使えないと思われた
        <- 預言者は殺される: 世界の変革者は既存勢力に排除される
      どのOSよりも柔軟で移植性に富み、性能の高く格安のOSがあると噂
        when. 1980 ~
        -> ビジネスや学問の世界でUNIXが受け入れられるようになった
          <- 大学で若い世代が先入観なしにUNIXに親しんだ
  オープンソース: UNIXが商標登録された故の代わりの呼称

## 1. UNIXの考え方: たくさんの登場人物たち
  タイムシェアリング, 時分割方式: 短時間に動作を切り替えて複数ジョブを同時並列に実行しているように見せる
    <-> バッチモード, 一括処理方式: 蓄えたジョブを順次実行
      大量のパンチカードが必要で処理に時間がかかった
  Multics: MITが開発した{タイムシェアリング}のOS
  スペース・トラベル: {Multics}システム上でのゲーム
  独自技術症候群: 他所で開発された技術を嫌い、すべて自前で用意しようとする
    自尊心による
    ex. よいプログラマ: 素晴らしいソフトウェアを書く
  UNIX: {スペース・トラベル}の動作環境として{Multics}を模したOS
    <- ケン・トンプソン: AT&T研究所で最初の{UNIX}をアセンブラで書いた
      when. 1969
      xe. 偉大なプログラマ: {よいプログラマ}の技術を盗み付加価値をつける
        -> `若年期`
      移植性のあるB言語で書き直し
        when. 1972

    when. 1973

## 2. 人類にとっての小さな一歩
  スモール・イズ・ビューティフル, 定理1: プログラムは小さく始めてそのままに保つ
    「これからは小型車の時代」というVW社のキャンペーン
    <- 問題の理解不足のために巨大な解決策を実装しようとしてしまう
      ex. ファイルコピー
        コピー元ファイル名を尋ね・存在チェック
        なければ通知
        コピー先ファイル名を尋ね・存在チェック
        あれば上書きするか尋ねる
        元ファイルを開く
        元ファイルが空だったら通知・中止
        コピー先を開く
        コピーする
        元/元のファイルを閉じる
    ex. 小型車の価値: 大型車を愛していたアメリカ人が原油価格高騰で小型車を見直下
      低燃費
      狭い駐車場に楽に駐車できる軽い乗り回し、応用力
      構造がシンプルでメンテしやすい
    -> メモリ消費量が少ない、軽い
    -> 理解しやすい
    -> 保守しやすい
      プログラムを書いた人がどうせ保守することになる
    -> 他のツールと組み合わせやすい
    -> 小さいプログラムは未来を予測し不測の事態に備えることを最初から諦める
      逆に時代遅れになることを避けられる
      <-> いろんなデータ形式に対応しようとする
        -> サポートされない形式が将来できて時代遅れになる
  -> １つのプログラムには１つのことをうまくやらせる, 定理2:
    ex. レイク・フライ: ナイル川の繁殖に全振りした12時間の寿命の食われまくっても繁殖に成功して雲を作る
    <-> しのびよる多機能主義: プログラマの創造性が単一目的外の機能を付加しようとしてしまう
      ファイルかコマンドラインからパラメータ入力すればいいのにダイアログを開発
      ASCIIを使えばいいのに特殊フォーマットを開発
      -> 車輪の再発明: すでに世の中にある車輪のようなものを再発明する無駄な作業
        <- `独自技術症候群`
    <-> lsコマンド: 使われもしない膨大なオプションや複数列への整列機能をもつ

## 3. 楽しみと実益をかねた早めの試作
  できるだけ早く試作する, 定理3: 1秒も無駄にせず大至急作れ、設計の完成を待つとかいいから
    <- 完璧な知識があれば不要なはずのテストが避けられない
    <- 仕様やマーケティングの変子は避けられない。客が本当に欲しいものは分からない
    <- 要求が完璧に分かっているならソフトウェアは要らない、ハードウェアでいい


  人間による3つのシステム: 人間が作るシステムは人間に似て若成老を時期を持つ
    第一のシステム, 若年期: 締切などで追い詰められて枝葉に構う余裕がなく重要な部分に集中して作ったシステム
      1人か数人で作られる
        <- {若年期}はそのシステム(失敗率80%越)の可能性に熱狂した人しか参加しない
      ハードコードされた汚い実装で高性能
      人々の想像力を刺激する面白いコンセプトを含む
        -> `第二のシステム`
    第二のシステム, 成熟期: 商業的に成功しがちでその成功に預かろうとする専門家が群がって作る
        <- `独自技術症候群`
        <-> `定理1`
        <-> `定理2`
      1つのことしかできなかった{第一のシステム}の反動であらゆる要求に応えようとして遅く巨大で欠陥だらけ
        -> 第三のシステム, 老年期: {成熟期}であらゆる可能性が試され、全て証明済みの枯れたコンセプトで見積もりも容易
          かつての熱狂は世の中の常識となり、もはや元の名前では呼ばれない
  伝統的設計: 優れた設計には熟慮が必要だという伝統的信念
    はじめから{第三のシステム}を目指すという不可能な目標
    -> 設計の90%が完成しないうちは1行たりともコードを書いてはならない
    <-> {UNIX}設計: UNIX開発者は3,4枚の短い機能仕様を書いてさっさとシステムを作り、必要ならドキュメントを書く
      これが正しい設計なのかは分かっていない、いや誰にも分からない、ものが出来上がるまでは
      満足するまでテストしては書き直す
      -> {第一のシステム}に最速で向かう
      <- {UNIX}は事前の設計なしで作成された
        <- `第一のシステム`
      熟慮が不要ということではなく、動くものが出来上がる前の細かい文書が無駄というだけ

## 4. 移植性の優先順位
  効率より移植性, 定理4: 効率は特定のハードへの最適化で実現されるので移植性とトレードオフだが移植性を選べ
    <- 使われる価値: 最も強力なコンピュータとは最もよく使われるコンピュータ
      -> スパコンでも強力なデスクトップでもなくノートパソコンこそ最強
    <- その効率化はハードウェアの性能向上を待てばタダで実現される
    <- Atariの移植性のない超効率的なソフトはゲーム機の変更で価値が暴落した
    <- Lotus 1-2-3の表計算ソフトはintel CPUに合わせて移植され価値を保った
      MSへの対応に遅れてExcelにとって変わられた
    -> 定理5: 数値データはASCIIフラットファイルに保存する
      バイナリ形式を禁止したデータの移植性
      パイプ
      <- ASCIIは最も一般的なデータ形式
      -> {UNIX}のあらゆる設定ファイルはASCIIで普通にテキスト編集できる


## 5. これこそてこの効果!
  所有: それの修正や使用などの許可などをコントロールできること
  付加価値: 他のソフトウェアを借用して独自拡張する
  梃子, 定理6: 既存ソフトウェアを活用する
    <- 基本的なロジックさえ書けないが、既存のコードの活用が得意で成功したプログラマ
    <- 競合相手のソフトの上位互換を作り直すのをやめて、それに{付加価値}を与えることでヒット商品を作れた例
      <- `独自技術症候群`
    ex. `偉大なプログラマ`

    模倣を防ぐのは不可能なので{所有}には意味がない
  {UNIX}はソースコードの{所有}を禁止した
    -> タダ同然で{UNIX}を手に入れることができた
      -> 開発費で苦しむハードメーカーが{UNIX}を使うようになり、{UNIX}が人気になった
        <- `使われる価値`
